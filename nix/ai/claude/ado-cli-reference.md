# Azure DevOps CLI Reference

The playbook for driving Azure DevOps from `az` + scripts (a lightweight replacement for the `u-ado-mcp` tools). PowerShell helpers live in `~/.claude/ado/`. This file and the scripts are nix-managed (`setup/nix/ai/claude/`).

## Organisations & projects (config files)

Each target is a JSON file at `~/.claude/ado/configs/<org>-<project>.json` (e.g. `urholm-Devkunt.json`). These are **generated by nix** from the `adoConfigs` list in `setup/nix/ai/claude/default.nix`. The scripts read them.

**Schema:**
```json
{
  "org": "urholm",
  "organizationUrl": "https://dev.azure.com/urholm",
  "project": "Devkunt",
  "process": "Devkunt Agile",
  "processBase": "Agile",
  "azureConfigDir": "/Users/<you>/git/urholm/.az",
  "repositories": ["devkunt-backend", "devkunt-frontend", "Devkunt.wiki"],
  "workItemTypes": ["Epic", "Feature", "User Story", "Task", "Bug"],
  "notes": "..."
}
```

**To add a target (the re-add path):** append an entry to `adoConfigs` in nix and rebuild — reproducible, preferred. If you must do it outside nix, hand-create `~/.claude/ado/configs/<org>-<project>.json` following the schema above (filename is always `<org>-<project>.json`).

**Load a config in a script / session:**
```bash
CFG=~/.claude/ado/configs/urholm-Devkunt.json
ORG=$(jq -r .organizationUrl "$CFG"); PROJ=$(jq -r .project "$CFG")
export AZURE_CONFIG_DIR=$(jq -r .azureConfigDir "$CFG")
```

## Authentication

Authenticate per machine with `az login` (browser or `--use-device-code`). Then set defaults:
```bash
az devops configure --defaults organization="$ORG" project="$PROJ"
```

**Gotchas (hard-won):**
- **MSA bearer bug (`TF400813`):** the `az devops` extension wraps Entra JWTs as `Authorization: Basic`, which **MSA-federated** identities (`live.com#…`) fail on. Use an **Entra-native** account for ADO automation, not an MSA.
- **Isolate identities on a shared machine:** to keep a project's ADO login separate from another `~/.azure` profile (e.g. a day-job tenant), run everything under an isolated config dir:
  ```bash
  export AZURE_CONFIG_DIR=/path/to/project/.az
  az login --allow-no-subscriptions      # then az devops configure --defaults …
  ```
- **Token shortcut** (for `az rest` / `curl` against REST APIs):
  ```bash
  TOKEN=$(az account get-access-token --resource 499b84ac-1321-427f-aa17-267ca6975798 --query accessToken -o tsv)
  ```

## Command flag rules

| Command | `--org` | `--project` | Notes |
|---------|---------|-------------|-------|
| `work-item create` | Yes | **Yes (required)** | create/delete/query need project |
| `work-item delete` | Yes | **Yes (required)** | needs both |
| `work-item update` | Yes | **No** | errors if `--project` passed |
| `work-item show` | Yes | **No** | resolves project from org + ID |
| `work-item relation add` | Yes | **No** | |
| `work-item relation list-type` | Yes | **No** | |
| `boards query` (WIQL) | Yes | **Yes (required)** | |
| `repos pr show` | Yes | **No** | resolves project from PR ID |
| `repos pr list` | Yes | Yes | also needs `--repository` |

## Relation types

Use display names: `Parent` (not `System.LinkTypes.Hierarchy-Reverse`), `Child`, `Related`, `Predecessor`, `Successor`. (`az boards work-item relation add --relation-type` accepts `parent`/`child`/`related`.)

## Fields

- Acceptance criteria: `Microsoft.VSTS.Common.AcceptanceCriteria`
- Value area: `Microsoft.VSTS.Common.ValueArea` (`Business`, `Architectural`)
- State values are process-dependent. Common: `New`, `Active`, `Resolved`, `Closed`.
- **Field names and allowed values vary by process template — discover via REST before use.**

## Work item type discovery (REST)

`az boards` has no WIT-metadata subcommand; use `az rest`.

```bash
# Types in a project
az rest --method get --uri "https://dev.azure.com/{org}/{project}/_apis/wit/workitemtypes?api-version=7.1" -o json
# Fields (with allowed values) for a type
az rest --method get --uri "https://dev.azure.com/{org}/{project}/_apis/wit/workitemtypes/{typeName}/fields?api-version=7.1&\$expand=allowedValues" -o json
# Process template of a project
az rest --method get --uri "https://dev.azure.com/{org}/_apis/projects/{project}/properties?api-version=7.1&keys=System.ProcessTemplateType" -o json
```

Template GUIDs: `6b724908-…`=Agile · `adcc42ab-…`=Scrum · `27450541-…`=CMMI · `b8a3a935-…`=Basic.

## show/update gotchas

- `-f` (fields) conflicts with default `--expand all`; use `--expand none` with `-f`, or omit `-f` and filter with `--query`.
- `--state` is a convenience flag on `update`; other fields use `--fields "Field.Name=value"`.

## Bulk operations

Parallelise:
```bash
echo "${IDS[@]}" | xargs -P 10 -I {} az boards work-item update --id {} \
  --state "Active" --fields "Microsoft.VSTS.Common.ValueArea=Architectural" \
  --org "$ORG" --query "id" -o tsv
```
When updating fields across a hierarchy, update ALL items, not just those matching a state filter.

## PR file discovery (MANDATORY)

**Never** use `git diff` between branches for PR scope — branch diffs include accumulated divergence. Use the iterations/changes REST API:
```bash
# Iterations (one per push)
curl -s -u ":$TOKEN" "https://dev.azure.com/{org}/{project}/_apis/git/repositories/{repoId}/pullRequests/{prId}/iterations?api-version=7.1"
# Changed files for an iteration — changeEntries[] is authoritative
curl -s -u ":$TOKEN" "https://dev.azure.com/{org}/{project}/_apis/git/repositories/{repoId}/pullRequests/{prId}/iterations/{iterationId}/changes?api-version=7.1"
```

## PR review threads (`az devops invoke`)

Inline comments need TWO top-level objects:
```json
{
  "comments": [{"parentCommentId": 0, "content": "...", "commentType": 1}],
  "threadContext": {
    "filePath": "/path/to/file",
    "rightFileStart": {"line": 42, "offset": 1},
    "rightFileEnd": {"line": 48, "offset": 1}
  },
  "pullRequestThreadContext": {
    "changeTrackingId": 5,
    "iterationContext": {"firstComparingIteration": 1, "secondComparingIteration": 1}
  },
  "status": 1
}
```
- `threadContext` anchors to code; without it the comment floats.
- **CRITICAL:** `filePath` + `rightFileStart`/`rightFileEnd` go in `threadContext`, NOT `pullRequestThreadContext` (silently fails there).
- `changeTrackingId` comes from the iteration-changes API. Delete a bad thread via `pullRequestThreadComments` with its `commentId`.
